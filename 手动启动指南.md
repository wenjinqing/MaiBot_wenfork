# 手动启动双机器人系统指南

## 架构说明

```
┌─────────────────────────────────────────────────────────┐
│                    消息转发架构                          │
└─────────────────────────────────────────────────────────┘

NapCat (君君, QQ:2477702109)
    ↓ WebSocket连接到 ws://127.0.0.1:8096
MaiBot-Napcat-Adapter/main.py (监听8096)
    ↓ WebSocket连接到 ws://127.0.0.1:8000
MaiBot/bot_junjun.py (君君机器人实例)


NapCat (伊伊, QQ:1033245881)
    ↓ WebSocket连接到 ws://127.0.0.1:8097
MaiBot-Napcat-Adapter-Yiyi/main.py (监听8097)
    ↓ WebSocket连接到 ws://127.0.0.1:8000
MaiBot/bot_yiyi.py (伊伊机器人实例)
```

**关键点**:
- 每个机器人有独立的适配器（main.py）
- 适配器统一转发消息到MaiBot主程序（端口8000）
- 两个机器人实例独立运行，互不干扰

## 启动顺序说明

**必须按照以下顺序启动**:

1. **先启动适配器** (main.py)
   - 适配器需要先监听端口
   - NapCat才能连接上来

2. **再启动机器人** (bot_junjun.py / bot_yiyi.py)
   - 机器人启动后会连接到适配器
   - 适配器转发消息给机器人

**错误的顺序会导致**:
- 机器人无法连接到适配器
- 消息无法转发
- 连接超时错误

## 常见问题

### Q1: 适配器启动失败，提示端口被占用

**症状**:
```
[ERROR] Address already in use: 8096
```

**解决**:
```bash
# Windows
netstat -ano | findstr "8096"
taskkill /PID <进程ID> /F

# Linux/Mac
lsof -i :8096
kill -9 <进程ID>
```

### Q2: 机器人无法连接到适配器

**症状**:
```
[ERROR] Connection refused to ws://127.0.0.1:8000
```

**解决**:
1. 确认适配器已启动
2. 检查端口配置是否正确
3. 检查防火墙设置

### Q3: NapCat无法连接到适配器

**症状**:
```
[ERROR] WebSocket connection failed
```

**解决**:
1. 确认适配器已启动并监听正确端口
2. 检查NapCat配置文件中的url
3. 检查网络连接

### Q4: 机器人不回复消息

**症状**: 发送消息后没有回复

**解决**:
1. **检查白名单配置**（最常见原因）
2. 检查适配器是否收到消息（查看日志）
3. 检查机器人是否处理消息（查看日志）
4. 检查talk_value设置

### Q5: 两个机器人回复混乱

**症状**: 君君回复了伊伊的消息，或反之

**解决**:
1. 确认两个适配器使用不同端口
2. 确认NapCat连接到正确的适配器
3. 检查配置文件中的QQ账号

ps aux | grep "main.py"

1. 停止机器人 (bot_*.py)
2. 停止适配器 (main.py)

## 优势说明

### 为什么使用独立的bot_*.py文件？

1. **配置隔离**: 每个机器人使用独立的配置文件
2. **日志清晰**: 日志中明确标识是哪个机器人
3. **独立重启**: 可以单独重启某个机器人
4. **调试方便**: 可以单独调试某个机器人

### 为什么适配器统一使用main.py？

1. **消息转发**: 适配器只负责转发消息，不需要区分机器人
2. **代码复用**: 两个适配器使用相同的代码
3. **配置简单**: 只需修改config.toml中的端口
